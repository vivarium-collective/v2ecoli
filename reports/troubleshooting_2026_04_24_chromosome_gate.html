v2ecoli chromosome replication gate & layer-4b architecture — findings

2026-04-24. Architecture investigation surfaced during BP1993 merge-prep review.

1. Gate issue found

The mechanistic-replisome gate at chromosome_replication.py:329-332 used strict equality: 

initiate_replication = not self.mechanistic_replisome or (
    np.all(n_replisome_trimers == 6 * n_oriC)
    and np.all(n_replisome_monomers == 2 * n_oriC)
)
Context: mechanistic_replisome defaults to False (permissive mode, matches LoadSimData's default). With the default config, the not self.mechanistic_replisome short-circuit makes the gate trivially True whenever mass threshold is crossed, and the == vs >= distinction doesn't matter — re-init fires on the Donachie criterion alone. This is why controlled multigen runs divide correctly in v2ecoli at the default config: the strict-equality gate was simply never being evaluated.

Once mechanistic_replisome=True is set (required for any analysis that wants to observe subunit competition — e.g., the Figure 2 uncontrolled-plasmid runs), the gate is evaluated, and the problem emerges. counts(states["bulk"], ...) in v2ecoli returns the raw total bulk count — chromosome and plasmid replication sit in layer 4b (generate.py:84-87), outside the main allocator's partition path. Against raw bulk counts routinely in the 10s–100s, a strict == against the exact request (12 trimers / 4 monomers for 2 oriCs) is effectively unsatisfiable — so the mechanistic gate is locked "off" any tick where subunit counts aren't precisely at the request value. In practice that is every tick, forever. Chromosome re-initiation never fires once mechanistic_replisome=True.

2. Controlled experiment — == gate with no plasmids, no contention

To isolate the gate-semantics problem from any plasmid or contention effects, we ran a single-generation mechanistic simulation with zero plasmids and the strict == gate restored (mechanistic_replisome=True, BP1993 irrelevant because no plasmids are present, cache rebuilt after regenerating the ParCa fixture via scripts/parca_run.py --mode full so all configs including ecoli-polypeptide-elongation are wired).

No plasmids, single generation, strict-equality == gate. (E) Chromosome active replisomes: the inherited round terminates at t≈22 min → 0, and stays at 0 for the remaining 20 min. No re-initiation event fires. (F) Cell mass crosses the 2-oriC threshold (1950 fg) at t≈29 min — the mass gate is satisfied, but the subunit gate isn't, because bulk subunit counts never precisely equal the per-request value of 12 trimers / 4 monomers.

This is the definitive test. With no plasmids at all, no competition for subunits, and all replisome subunits abundant (pol_core≈500, β-clamp≈200, etc.), the == gate still prevents chromosome re-initiation. The gate is mathematically unsatisfiable against raw bulk counts whenever the per-tick subunit count isn't precisely at the request value — which is almost always. The cell divides at t = 42.0 min on mass-trigger alone, but with 0 active replisomes — a broken lineage state from which gen 2 would start stuck in the same condition.

Under vEcoli's allocator, the three-phase Requester → Allocator → Evolver pattern would deliver exactly the requested 12 trimers / 4 monomers to chromosome — so == would pass against the allocator-partitioned values. That's why the == operator was originally chosen in the code. When the replication processes were moved out of the allocator into the standalone layer-4b simplification, the operator should have been switched to ≥ to reflect the new semantic (raw bulk reads rather than partitioned deliveries). The corrected operator finally does that.

3. The patch

Changed == to >= at chromosome_replication.py:329-332 so the gate matches its own documented intent ("there are enough replisome subunits"): 

initiate_replication = not self.mechanistic_replisome or (
    np.all(n_replisome_trimers >= 6 * n_oriC)
    and np.all(n_replisome_monomers >= 2 * n_oriC)
)

Validation: a controlled mechanistic single-gen run shows chromosome re-initiation firing within 1 fg of the mass threshold (cell_mass = 1950.6 vs threshold 2 × 975 = 1950 fg), producing the expected n_active_replisomes: 0 → 4 transition and division on schedule.

4. Post-patch behavior under uncontrolled plasmid load (1 seed, BP1993 off)

(A) Plasmid copy number 1 → 3254 across 42 min — runaway reproduces cleanly. (B) Chromosome active replisomes: inherited round terminates at t ≈ 22 min, then re-initiation fires cleanly at t ≈ 30 min (0 → 4) the moment cell_mass crosses n_oriC × 975 = 1950 fg. (C) DnaG oscillates 0↔8 on a ~6-tick sawtooth (plasmid-replisome round-time) rather than sustained depletion; any mass-threshold crossing will eventually coincide with a DnaG crest and let the chromosome gate pass.

5. The chromosome re-init is an artifact of sequential execution — not competition

Layer 4b in generate.py:84-87 is a flat list: 

['ecoli-complexation', 'ecoli-chromosome-replication',
 'ecoli-plasmid-replication',
 'ecoli-polypeptide-initiation', 'ecoli-transcript-initiation']

No allocator, no ReconciledStep, no partitioning between these processes. In process-bigraph semantics this means they execute sequentially in listed order, each one reading the bulk state left behind by the previous one. Chromosome replication runs before plasmid replication every tick.

Chromosome and plasmid replication are not competing with each other in v2ecoli. They are two independent processes running back-to-back on the same state. Chromosome reads raw bulk at tick start, takes what its gate asks for, commits its delta. Plasmid then reads the already-reduced bulk and takes what's left. This is an implicit priority ordering — determined by the order processes are listed in layer 4b — not a model of resource competition. Chromosome wins every tick by construction, not by biology.

Empirical confirmation from the t=1769→1770 transition (DnaG spike from a plasmid-replisome termination release, coincident with the Donachie mass-threshold crossing):

Had layer 4b listed plasmid before chromosome, the outcome would invert: plasmid would have taken all 8 DnaG, chromosome would have seen 0, and re-init would never have fired. The outcome is purely list-order, not subunit arithmetic.

6. Comparison with vEcoli mechanistic behaviour (under allocator)

Representative mechanistic phenotype under vEcoli-style allocator-based resource partitioning: plasmid runaway (A, 1 → ~4000 across 10 seeds), chromosome active replisomes drop to 0 after the inherited round terminates and never re-initiate (C), and DnaG drops to near-zero while the other five replisome subunits stay abundant (E). The arrest-and-DnaG- depletion signature is exactly what a proportional-fairness allocator produces in this regime.

In vEcoli, partitioned processes compete for bulk via the Requester → Allocator → Evolver pattern: each process submits a request, and the allocator distributes available bulk proportionally to total demand. Under uncontrolled plasmid load, chromosome's request for 4 DnaG is dwarfed by plasmid's ~2000 — so chromosome receives (4 / 2004) × 8 ≈ 0.016 DnaG, well below the mechanistic gate threshold, and never re-initiates. That is the biology this panel is really testing: what does mechanistic competition between chromosome and plasmid replication look like when subunits are fairly partitioned? The answer is sustained chromosome arrest under runaway plasmid load.

Under post-patch v2ecoli, chromosome re-initiation fires anyway (§4-§5) because layer 4b's sequential-priority execution never forces chromosome to compete proportionally — it simply takes what it wants before plasmid runs. To reproduce this vEcoli allocator-driven phenotype in v2ecoli, the reconciled-replication refactor (§8) is needed.

7. Competition semantics: vEcoli vs. v2ecoli

vEcoli models real competition with the Requester → Allocator → Evolver pattern on PartitionedProcess instances. Each process submits a request; the allocator partitions the available bulk proportionally to total demand. Under uncontrolled plasmid load:

ArchitectureDnaG allocation with chromosome demand ≈ 4, plasmid demand ≈ 2000, available = 8
vEcoli partitioned + allocator Chromosome share = (4 / 2004) × 8 ≈ 0.016 DnaG. Below gate threshold → chromosome fails → arrest.
v2ecoli layer-4b sequential Chromosome (listed first) takes 4 DnaG outright. Plasmid takes remaining 4. Chromosome wins every time.

These are fundamentally different mechanisms. vEcoli models competition (fairness-weighted shares). v2ecoli models priority (first-in-list-wins). The outputs can look similar or wildly different depending on the regime; they're not interchangeable.

8. Path forward — reconciled replication layer

v2ecoli already ships a ReconciledStep (v2ecoli/steps/reconciled.py) that implements the same proportional-fairness math as vEcoli's allocator, packaged as a single Step instead of the three-step Requester/Allocator/Evolver chain. It's currently used for RNA degradation (reconciled_2) and elongation (reconciled_3) — but not replication. generate_reconciled.py doesn't even import PlasmidReplication.

Blocker: ReconciledStep asserts isinstance(p, PartitionedProcess) at line 176. Both replication processes are currently plain Steps: ChromosomeReplication(Step) and PlasmidReplication(Step). They do not expose the calculate_request() / evolve_state() split that ReconciledStep needs. They'd need to be refactored back into PartitionedProcess form before they can be wired into a reconciled layer.

Required steps to enable reconciled replication:

  1. Refactor ChromosomeReplication and PlasmidReplication from Step back to PartitionedProcess, splitting the current update() into calculate_request(timestep, states) (what subunits does each process want?) and evolve_state(timestep, proc_states) (given what was allocated, do the work). Plasmid's BP1993 ODE integration in _prepare() needs to be placed carefully — it's stateful and reads/writes plasmid_rna_control.
  2. Create a new reconciled_replication layer in generate_reconciled.py that groups chromosome and plasmid replication. Add the missing PlasmidReplication import.
  3. Validate: controlled regime should still divide to ~28 cpc; uncontrolled regime under reconciled layer should now model genuine DnaG competition — chromosome proportionally starved when plasmid demand dominates.

Estimated effort: 2–4 days of careful work. Notes captured in memory project_v2ecoli_reconciled_replication_task.md.

9. Separate issue — DnaG-limiting dynamics across generations

Distinct from the gate-semantics problem above, this section documents a DnaG-supply observation surfaced while validating the patched gate across generations. It is unrelated to the ==/≥ architecture fix — the behavior shows up even under the corrected ≥ gate. Flagged separately because it touches metabolism.aa_enzymes-dependent protein synthesis rates and DnaG expression calibration, not layer-4b partitioning.

No-plasmid lineage, 7 generations, patched ≥ gate. (D) Chromosome active replisomes across the lineage — gens 1–4 follow the normal 2→0→4 pattern (termination, then re-init post-mass-threshold), gen 5 fails to re-initiate (inherited round terminates, no new round fires before division), gens 6–7 then adaptively downshift to 1-oriC mode (daughter of gen 5 has 1 chromosome and 0 forks; mass is already above the 1-oriC threshold of 975 fg at birth, so re-init fires immediately at t≈1s each gen). Cell mass (lower panel) shows the adaptation: peaks cluster around the 2-oriC threshold for gens 1–5 and around the 1-oriC threshold for gens 6–7.

DnaG bulk is 0 across every tick of every generation — all available DnaG is sequestered in bound replisomes. Re-init only fires when a transient DnaG synthesis event releases a molecule that the chromosome gate catches in the same tick. Gens 1–4 squeak through; gen 5 misses; gens 6–7 succeed because the 1-oriC regime needs only 2 × 1 = 2 DnaG rather than 2 × 2 = 4, which is stochastically easier to satisfy.

This matches the spirit of the submitted proposal's claim that "DnaG is not maintained at the level required for re-initiation across multiple generations even in the absence of plasmids" — the first failure emerges at gen 5 — but the lineage does not collapse. It adapts down to a lower-oriC regime where the mechanistic-subunit demand is smaller. A genuine lineage collapse would need either sustained 1-oriC-insufficiency or a further perturbation.

Implication for Aim 1a: DnaG expression and turnover recalibration is still a legitimate item — not because the gate was broken (that's fixed), but because DnaG bulk levels leave essentially no headroom above the mechanistic gate threshold, making re-init stochastically fragile. Tuning DnaG synthesis rate against quantitative-proteomics references (e.g., a 2–3× increase consistent with literature measurements of primase abundance) would restore robust 2-oriC multigen dynamics without requiring architectural changes.
