Chromosome replication refactor: Step → PartitionedProcess

2026-04-24. Wires chromosome replication into v2ecoli's allocator-based partitioning machinery so it can compete fairly with plasmid replication (once plasmid is similarly refactored). Behavior on the chromosome-only path is preserved.

1. Why this refactor

v2ecoli's chromosome replication has been a plain Step sitting in execution layer 4b alongside complexation and the various initiation processes. That made sense for the chromosome-only scope — chromosome replication is the only consumer of replisome-specific subunits (DnaG, DnaB, HolA, pol-III core, β-clamp), so there was nothing for it to compete with.

That changes when plasmid replication is added. Both replication processes draw from the same replisome subunit pool, and under uncontrolled plasmid load DnaG can be drained by plasmid replisomes faster than the chromosome process can claim it. To capture this competition correctly we need the two processes to share an allocator that proportionally partitions subunit availability between them.

v2ecoli already implements that mechanism — the Requester / Allocator / Evolver pattern in v2ecoli/steps/partition.py, used today for RNA degradation and the elongation processes. Eligibility for that mechanism requires inheriting from PartitionedProcess (which exposes calculate_request + evolve_state) rather than plain Step. So step one is to refactor chromosome replication into the partitioned-process form. Plasmid replication will follow as a separate refactor, joining chromosome in the same allocator layer.

2. What changed

2.1 Class refactor — v2ecoli/processes/chromosome_replication.py

  • Base class: EcoliStep → PartitionedProcess (imported from v2ecoli.steps.partition).
  • The single update(states, interval) entry point is removed; the framework's PartitionedProcess.update takes over and dispatches to the two-phase methods.
  • The biology code that used to live in _prepare(states) is now calculate_request(timestep, states). It no longer discards the request dict — it returns it so the framework can apply the requested allocation before the evolve step.
  • The biology code that used to live in _evolve(states) is now evolve_state(timestep, states). The body is unchanged modulo the renamed signature.

No biology code was rewritten. All gate logic, replisome accounting, fork advancement, termination, and daughter spawning are byte-identical to the previous implementation; the refactor only restructures the class around the partitioned-process interface.

2.2 Wiring in v2ecoli/generate.py

Chromosome replication moves from the standalone-step path to the partitioned-process path. The choice of allocator layer follows v2ecoli main's flow definition (see configs/default.json): chromosome replication has the same ecoli-tf-binding dependency depth as ecoli-rna-degradation, so they share the allocator_2 layer. Specifically:

v2ecoli/generate_reconciled.py received matching edits so the reconciled architecture also picks up the refactored chromosome class via its imported ALLOCATOR_LAYERS dict.

2.3 What did NOT change

  • Plasmid replication still inherits from Step; plasmid refactor is a separate follow-up commit.
  • Default execution semantics for chromosome-only sims: with a single process in allocator_2 (when run without plasmids), the allocator math reduces to a pass-through — chromosome receives its full request whenever supply is sufficient. Functionally identical to the previous Step behavior; differs only by going through three Step dispatches per tick (Requester → Allocator → Evolver) instead of one.
  • No biology constants or gate operators changed.

4. Diagnosis of the pre-existing test failure

The error surfaces in the bigraph-schema realize step on the post-division daughter cell. The chromosome / division portion of the pipeline runs correctly: 

DIVISION at t=2483s (dry_mass=702.0 fg, threshold=702.0 fg, chromosomes=2)
  DAUGHTERS: 00 (bulk=...) + 01 (bulk=...)

After the daughter cells are constructed, the framework attempts to realize their state against the schema and fails on the listeners.growth_limits sub-tree: 

ValueError: realize Array at path=('agents', '00', 'listeners', 'growth_limits'):
np.array failed. encode type=list, len=39, first_row_type=ndarray,
first_row_len=0, dtype=float64. Original: setting an array element with a
sequence. The requested array has an inhomogeneous shape after 1 dimensions.
The detected shape was (39,) + inhomogeneous part.

4.1 Why parent cells don't hit this

On a parent cell, every simulation tick the polypeptide_elongation process writes all 36 fields of growth_limits with shape-consistent values (amino-acid-shaped arrays of length 21, scalars as floats, etc.). By the time the schema realize step inspects the parent's growth_limits, every field has a real, written-by-the-tick value with consistent shape. np.array(...) over those values succeeds.

4.2 Why daughter cells DO hit it

On a freshly divided daughter cell, no process has yet written growth_limits. The fields are populated from the schema's declared defaults, which are deliberately heterogeneous: empty arrays for the array-typed fields ('overwrite[array[float[uM]]]' → _default: []), and zero scalars for the scalar-typed fields ('overwrite[float[uM]]' → _default: 0.0). The dict therefore contains a mix of np.array([]), 0.0, etc.

The realize-Array dispatch in bigraph_schema/methods/realize.py handles dict inputs by flattening to a list of values: 

elif isinstance(encode, dict):
    encode = dict_values(encode)
...
state = np.array(encode, dtype=schema._data)

With 39 mixed-shape values in the list, np.array(...) raises an inhomogeneous-shape ValueError. The error message reports len=39, first_row_type=ndarray, first_row_len=0 — exactly the empty-array default for the first listed field, with subsequent fields having different shapes.

Root cause is in the schema framework, not in v2ecoli. growth_limits is declared in the process schema as a struct of named typed fields. The realize dispatch is treating it as an Array and flattening the dict — which only works when the parent's written values happen to share a uniform shape. Default-construction inevitably surfaces the heterogeneity and fails.

5. Proposed workaround

The cleanest local fix lives in v2ecoli/library/division.py. divide_cell currently builds daughter states with bulk, unique, environment, boundary, and process_state populated, but deliberately does not copy listeners. Adding a listener copy in the same style as the existing environment / boundary copies would side-step the schema-realize bug: 

if 'listeners' in cell_state:
    d1_state['listeners'] = copy.deepcopy(cell_state['listeners'])
    d2_state['listeners'] = copy.deepcopy(cell_state['listeners'])

Each daughter inherits the parent's last-tick listener values — i.e., ndarrays + scalars with the parent's actual consistent shapes. The realize step succeeds, the daughter runs normally, and on its first post-division tick the listener fields are overwritten by their writing processes (every listener field is declared as overwrite[...], which fully replaces the value rather than accumulating). The inheritance is therefore load-bearing for the framework but biology-neutral.

This workaround is intentionally not applied in this commit — it is a fix for a pre-existing failure that was inherited from the main branch, and is best landed as its own targeted commit (or upstream in bigraph-schema) so that it can be reasoned about independently of the chromosome refactor.

5.1 The proper fix

The proper fix belongs in bigraph-schema's realize-Array dispatch. When the encode value is a dict and the schema describes a struct of named fields (rather than a single contiguous array), realize should recurse into each field with that field's specific schema rather than collecting the dict's values and trying to materialize them as a single ndarray. The current dispatch happens to work for dicts whose values are shape-uniform because np.array(list_of_uniform_arrays) succeeds; it does not handle struct-style heterogeneity. Filing this as an upstream issue or PR is the durable fix; the v2ecoli-side workaround above is appropriate until then.
