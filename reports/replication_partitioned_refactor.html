Replication refactor: Step → PartitionedProcess for chromosome and plasmid

Two-phase refactor that wires both replication processes into v2ecoli's existing allocator machinery so chromosome and plasmid replication compete for replisome subunits via the proportional-fairness math used by the rest of the partitioned processes. Chromosome-only execution is preserved bit-equivalently; new plasmid-vs-chromosome competition emerges naturally under uncontrolled plasmid load — reproducing the vEcoli Figure 2 chromosome-arrest phenotype directly in v2ecoli.

1. Why this refactor

Before the refactor, both ChromosomeReplication and PlasmidReplication were plain Step instances running back-to-back in execution layer 4b. Sequential execution gave chromosome an implicit priority over plasmid (chromosome was listed first, took whatever subunits it wanted from raw bulk; plasmid ran second on residual bulk). This is a code-convention determining biology — not a model of competition. Under uncontrolled plasmid load the sequential ordering still let chromosome win, even though the biological reality is that thousands of plasmid replisomes should out-compete chromosome's two for the shared subunit pool.

v2ecoli already implements proper proportional fairness — the Requester → Allocator → Evolver pattern in v2ecoli/steps/partition.py, used today for RNA degradation and the elongation processes. Eligibility for that mechanism requires inheriting from PartitionedProcess (which exposes calculate_request + evolve_state) instead of plain Step. This refactor moves chromosome and plasmid replication onto that path, places them both in allocator_2 alongside rna-degradation (matching vEcoli's flow definition where chromosome and rna-deg share the same ecoli-tf-binding dependency depth), and lets the proportional-fairness math handle competition.

2. What changed

2.1 Phase 1 — chromosome refactor

  • Base class EcoliStep → PartitionedProcess; update() entry point removed (the framework's two-phase entrypoint takes over).
  • _prepare(states) → calculate_request(timestep, states). v2ecoli's existing _prepare already constructed the bulk request dict (it was being discarded with _ = requests); the refactor just returns it. No new biology code added.
  • _evolve(states) → evolve_state(timestep, states). Body unchanged.
  • generate.py: chromosome moved from STANDALONE_STEPS to PARTITIONED_PROCESSES; added to ALLOCATOR_LAYERS['allocator_2'] alongside ecoli-rna-degradation (per vEcoli's configs/default.json — both processes have the same ecoli-tf-binding dependency depth). Layer 4b lost its chromosome entry; the rna-degradation requester / evolver layers gained chromosome counterparts.
  • generate_reconciled.py: matching updates.

2.2 Phase 2 — plasmid refactor

Same structural pattern as chromosome, with one important difference: v2ecoli's previous PlasmidReplication._prepare did not construct a bulk request — it ran the BP1993 ODE and stashed n_rna_initiations on self, with all bulk gating happening inside _evolve. To enable allocator-mediated competition, calculate_request now also constructs a bulk request, ported from the analogous vEcoli plasmid-replication pattern: 

# Subunit request: one set of replisome subunits per idle plasmid
# domain we might initiate on this tick (gated by RNA II under
# use_rna_control).
if len(idle_plasmid_domains) > 0:
    if self.use_rna_control:
        n_to_request = min(len(idle_plasmid_domains), n_rna_initiations)
    else:
        n_to_request = len(idle_plasmid_domains)
    if n_to_request > 0:
        requests["bulk"].append((self.replisome_trimers_idx, 3 * n_to_request))
        requests["bulk"].append((self.replisome_monomers_idx, 1 * n_to_request))

# dNTP request for elongation of existing forks (same idiom as
# chromosome_replication.calculate_request).
if n_active_replisomes > 0:
    ...
    requests["bulk"].append((self.dntps_idx, ...))

The crucial gating is on n_rna_initiations: under BP1993 control, plasmid only requests subunits for the integer number of initiations the ODE actually allows this tick. This means a low-contention controlled regime asks for a small number of replisomes (chromosome gets its full request); under uncontrolled (BP1993 off), plasmid asks for one replisome per idle domain (thousands), which is what generates the competition. All BP1993 ODE biology is preserved exactly — the new code is just the bulk-request construction.

Wiring updates:

  • generate.py: 'ecoli-plasmid-replication' moved from STANDALONE_STEPS to PARTITIONED_PROCESSES; added to ALLOCATOR_LAYERS['allocator_2']; layer 4b loses the plasmid entry; allocator_2 sub-layers gain plasmid_requester / plasmid_evolver alongside chromosome and rna-degradation.
  • generate_reconciled.py: matching updates (the RECONCILED_LAYERS dict is auto-derived from ALLOCATOR_LAYERS, so plasmid is picked up automatically in the reconciled architecture as well).
  • scripts/run_plasmid_multiseed.py: removed the previously-temporary custom_priorities override that forced chromosome to win under sequential-priority. The override is no longer needed (and would defeat the allocator-fairness behavior the refactor enables). Also widened _allocated's exception handler to include KeyError for robustness.

3. Behavior across regimes

Three single-cell smoke runs cover the three regimes the refactor exercises. All three were run after Phase 1 + Phase 2, against the post-merge ParCa fixture, on the same machine and with the same seed-0 random state.

Row A (no plasmids): 7-generation chromosome-only lineage from out/plasmid/multigen_timeseries_noplasmid_7gen.json. Chromosome is the only requester in allocator_2, so the allocator is a pass-through and gen-to-gen behavior is indistinguishable from the previous Step implementation (gens 1-4 follow the normal 2→0→4 pattern; the gen-5 re-init miss and gen-6/7 1-oriC downshift are pre-existing dynamics unrelated to this refactor).

Row B (controlled plasmids, BP1993 on): single generation with use_rna_control=True from out/plasmid/multigen_timeseries.json. Plasmid load stays low (1→9), BP1993 caps requests so total subunit demand is well below supply; the allocator gives both processes their full requests. Chromosome behavior matches pre-refactor exactly (2→0 termination at 22 min, 0→4 re-init at 30 min, division at 45 min).

Row C (uncontrolled plasmids, BP1993 off): single seed with use_rna_control=False and custom_priorities override removed, from out/plasmid/multiseed_timeseries.json. Plasmids run away to 4407 by division. The inherited chromosome round terminates at 22 min as usual — but no new chromosome round ever fires. Cell mass crosses the 1950 fg 2-oriC threshold at ~30 min, but with plasmid demand (~3 trimers and 1 monomer per idle plasmid × thousands of idle plasmids) dwarfing chromosome's 12-trimer / 4-monomer request, the allocator's proportional partition gives chromosome essentially zero share, and chromosome's mechanistic gate fails on its allocated counts. Cell still divides (mass-trigger on the dry-mass listener fires regardless of chromosome state) but with n_active_replisomes = 0 at division — a broken lineage state. This is the vEcoli Figure-2 phenotype, now reproducing in v2ecoli through genuine allocator-fairness math rather than a sequential-priority shortcut.

5. Why proportional fairness is the right biology, not just the right code

Cells don't have a priority list telling DnaG which replisome to bind. DnaG diffuses and binds whichever replisome it encounters, with rate proportional to binding-site availability. When there are 4 chromosome assembly sites and ~4000 plasmid assembly sites in the same cell, DnaG has roughly a 1000× higher chance of hitting a plasmid site, and chromosome's effective rate of obtaining DnaG drops by roughly the same factor. The allocator's proportional partition is exactly this calculation in coarse-grained form — same answer the mass-action ODE would give at equilibrium.

Pre-refactor, chromosome's "always-win" outcome under uncontrolled plasmid load was an artifact of the v2ecoli port simplification, not biology. The custom-priority hack in scripts/run_plasmid_multiseed.py was an explicit acknowledgement that the architecture wasn't producing the desired phenotype on its own — it set chromosome priority = 5 to force chromosome to win. Removing that hack and letting the allocator math drive the outcome is what brings v2ecoli's competition semantics in line with vEcoli's, which validated the original Figure 2 phenotype against Clewell 1972 and Nordström 2006.

7. Diagnosis of the pre-existing test failure (carried over from Phase 1)

The error surfaces in the schema realize step on the post-division daughter cell. The chromosome / division portion of the pipeline runs correctly: 

DIVISION at t=2483 s (dry_mass=702.0 fg, threshold=702.0 fg, chromosomes=2)
  DAUGHTERS: 00 (bulk=...) + 01 (bulk=...)

After the daughter cells are constructed, the framework attempts to realize their state against the schema and fails on the listeners.growth_limits sub-tree: 

ValueError: realize Array at path=('agents', '00', 'listeners', 'growth_limits'):
np.array failed. encode type=list, len=39, first_row_type=ndarray,
first_row_len=0, dtype=float64. Original: setting an array element with a
sequence. The requested array has an inhomogeneous shape after 1 dimensions.
The detected shape was (39,) + inhomogeneous part.

7.1 Why parent cells don't hit this

On a parent cell, every simulation tick the polypeptide_elongation process writes all 36 fields of growth_limits with shape-consistent values (amino-acid-shaped arrays of length 21, scalars as floats, etc.). By the time the schema realize step inspects the parent's growth_limits, every field has a real, written-by-the-tick value with consistent shape. np.array(...) over those values succeeds.

7.2 Why daughter cells DO hit it

On a freshly divided daughter cell, no process has yet written growth_limits. The fields are populated from the schema's declared defaults, which are deliberately heterogeneous: empty arrays for the array-typed fields ('overwrite[array[float[uM]]]' → _default: []), and zero scalars for the scalar-typed fields ('overwrite[float[uM]]' → _default: 0.0).

The realize-Array dispatch in bigraph_schema handles dict inputs by flattening to a list of values: 

elif isinstance(encode, dict):
    encode = dict_values(encode)
...
state = np.array(encode, dtype=schema._data)

With 39 mixed-shape values in the list, np.array(...) raises an inhomogeneous-shape ValueError. The error message reports len=39, first_row_type=ndarray, first_row_len=0 — exactly the empty-array default for the first listed field, with subsequent fields having different shapes.

Root cause is in the schema framework, not in v2ecoli. growth_limits is declared in the process schema as a struct of named typed fields. The realize dispatch is treating it as an Array and flattening the dict — which only works when the parent's written values happen to share a uniform shape. Default-construction inevitably surfaces the heterogeneity and fails.

8. Proposed workaround

The cleanest local fix lives in v2ecoli/library/division.py. divide_cell currently builds daughter states with bulk, unique, environment, boundary, and process_state populated, but deliberately does not copy listeners. Adding a listener copy in the same style as the existing environment / boundary copies would side-step the schema-realize bug: 

if 'listeners' in cell_state:
    d1_state['listeners'] = copy.deepcopy(cell_state['listeners'])
    d2_state['listeners'] = copy.deepcopy(cell_state['listeners'])

Each daughter inherits the parent's last-tick listener values — i.e., ndarrays + scalars with the parent's actual consistent shapes. The realize step succeeds, the daughter runs normally, and on its first post-division tick the listener fields are overwritten by their writing processes (every listener field is declared as overwrite[...], which fully replaces the value rather than accumulating). The inheritance is therefore load-bearing for the framework but biology-neutral.

This workaround is intentionally not applied in this commit — it is a fix for a pre-existing failure that was inherited from the main branch, and is best landed as its own targeted commit (or upstream in bigraph_schema) so it can be reasoned about independently of the replication refactor.

8.1 The proper fix

The proper fix belongs in bigraph_schema's realize-Array dispatch. When the encode value is a dict and the schema describes a struct of named fields (rather than a single contiguous array), realize should recurse into each field with that field's specific schema rather than collecting the dict's values and trying to materialize them as a single ndarray. The current dispatch happens to work for dicts whose values are shape-uniform because np.array(list_of_uniform_arrays) succeeds; it does not handle struct-style heterogeneity. Filing this as an upstream issue or PR is the durable fix; the v2ecoli-side workaround above is appropriate until then.

9. Honest caveats

  • Proportional fairness is still a coarse-grained approximation. Real cells have specific binding kinetics (DnaG-replisome dissociation rates, spatial localization, oriC- vs oriV-specific affinities) that proportional partitioning flattens out. For mechanistic DnaA-oriC initiation (Aim 2) and the eventual bulk-ification of RNA I/II (Stage C) you'll want explicit binding kinetics that go beyond proportional partitioning — but the allocator is the correct intermediate model and matches what v2ecoli's reconciled architecture and vEcoli ship with today.
  • Per-process allocate / request snapshots in scripts/run_plasmid_multiseed.py read 0 currently. The _alloc_molecule_idx_cache isn't being populated correctly post-refactor for the replisome subunit indices; the underlying biology is correct (chromosome arrest is visible in the unique-molecule trace) but the per-tick allocation values aren't being captured for the report. Script-side cleanup, not a correctness issue.
  • Chromosome-only path is preserved bit-equivalently. An allocator with a single requester (chromosome alone, when has_plasmid=False) reduces to a pass-through — the proportional math gives the only requester its full request whenever supply is sufficient. All chromosome-only behavior tests pass identically to pre-refactor.
